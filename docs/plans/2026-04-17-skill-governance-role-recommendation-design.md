# Skill Studio 权限挂载助手：AI 推荐角色 list + package 视图改造

日期：2026-04-17
状态：已确认，进入实现
范围：`Skill Studio` 权限快捷挂载助手

## 1. 背景

当前 `Skill Studio` 权限挂载助手存在三个核心偏差：

1. 入口错位：系统先让用户手填 `组织路径 / 岗位 / 职级`，而不是先基于 `Skill` 内容与 `org memory` 推荐角色 list。
2. 交互低效：部门、岗位、职级缺少搜索弹选与批量添加能力。
3. 模型不对：用户想确认的是“一个角色下面整套权限 package”，而不是零散地维护一条条岗位文本。

## 2. 目标

如果用户打开权限挂载助手，则系统应先完成“AI 推荐角色 list”，再让用户确认或校正，最后基于确认后的角色 list 生成和查看权限 package。

本期目标：

- 将“手填岗位”改成“AI 推荐角色 list + 人工校正”
- 推荐时同时读取 `Skill` 内容与 `org memory`
- 如果推荐置信度不足，则降级到“编辑人部门角色”
- 补齐部门 / 岗位 / 职级的搜索弹选与部门批量添加
- 增加角色 list 视图与 package 汇总视图

## 3. 确认后的产品定义

### 3.1 角色推荐入口

- 如果进入权限挂载助手，则第一屏先展示 `AI 推荐角色 list`
- 如果角色 list 未确认，则只展示预估影响，不直接进入完整 package 编辑
- 如果角色 list 已确认，则写入现有 `service-roles`，并将下游策略标记为待刷新

### 3.2 推荐输入源

推荐引擎同时读取以下信号：

- `Skill`：名称、描述、`system_prompt`、绑定资产、数据查询、输出对象
- `org memory`：部门职责、岗位职责、流程、OKR / 绩效语义
- 当前用户：编辑人部门、岗位

### 3.3 置信度与降级

- 高置信度：`Skill` 与 `org memory` 指向同一批角色
- 中置信度：有交集，但证据不足，需要确认
- 低置信度：没有稳定交集，则默认降级到编辑人部门角色

### 3.4 package 视图

package 不新建底层实体，而是复用现有结构做聚合展示：

- 表字段可见性 package
- 知识库遮蔽等级 package
- 资产挂载 package

## 4. 实现策略

### 4.1 本期不改后端主模型

本期不新增新的 `role_package` 持久化模型，仍复用：

- `service-roles`
- `role-asset-policies`
- `mount-context`
- `mounted-permissions`

### 4.2 新增前端推荐引擎

新增一个本地推荐引擎模块，输入：

- `SkillDetail`
- `BoundAssetItem[]`
- `OrgMemorySnapshot[]`
- `Department[]`
- `Position[]`
- 当前用户

输出：

- 推荐角色列表
- 每个角色的置信度
- `Skill` 依据
- `Org Memory` 依据
- 是否为低置信度降级结果

### 4.3 交互改造

在现有治理侧栏中增加三块内容：

1. `AI 推荐角色 list`
2. `服务岗位（含搜索校正）`
3. `角色 package 总览`

## 5. 前端变更点

### 5.1 `SkillGovernancePanel`

- 加载 `org-memory snapshots`
- 加载部门和岗位列表
- 引入当前登录用户
- 基于上下文计算推荐角色 list
- 支持“一键确认推荐角色 list”

### 5.2 `SkillGovernanceCards`

- 新增推荐角色卡片
- 为服务岗位卡片增加搜索弹选与部门批量添加
- 新增角色 package 总览卡片

### 5.3 测试

- 新增推荐引擎纯逻辑测试
- 扩展治理面板测试，覆盖：
  - 推荐角色展示
  - 低置信度降级
  - 推荐角色确认写入 `service-roles`

## 6. 非目标

本期先不做：

- 新的后端 `role_package` 实体
- 真正按角色拆分的知识库遮蔽写回接口
- 完整的后端 AI 推荐服务

这些能力后续可以在当前 UI 和前端推荐引擎之上继续下沉。

## 7. 验收标准

- 如果 `Skill` 与 `org memory` 有明显交集，则能先看到推荐角色 list
- 如果推荐置信度不足，则能降级到编辑人部门角色
- 如果用户要校正，则部门 / 岗位 / 职级都可搜索后添加
- 如果用户选择某个部门，则可批量加入该部门岗位
- 如果用户确认角色 list，则调用现有 `service-roles` 保存链路
- 如果角色 list 改变，则声明与测试集重新标记为 stale
